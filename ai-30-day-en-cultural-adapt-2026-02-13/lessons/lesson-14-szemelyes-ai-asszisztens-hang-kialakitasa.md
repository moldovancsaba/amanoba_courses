# Lesson 14: Creating a personal "AI assistant" voice **In a sentence:** A personal AI assistant is good because it always brings the same quality.
**Time:** 20-30 minutes
**Tangible output:** Role-based system instruction

## Learning goal

Handling several types of tasks in a uniform tone. In this situation, you are the office manager and you have to make a quick but reliable decision.
By the end of today's practical task, you will have role-based system instructions.
Key insight: assistant system request starts working when you make the output measurable at start-up. At the end of the lesson, you will be able to: **set up an AI assistant of constant quality**

- [ ] At least one measurable indicator is attached to the output: **consistent output for 4 task types**.
- [ ] There is a separate control step to deal with the main risk: **inconsistent response quality**.

**Primary role:** office manager
**Secondary Role:** Team Assistant

## What

### What is this?

This lesson shows in practice how to use the **assistant system request** method in a real work situation.
The goal is not a theory, but an output that you can really use in your next working day.

### What isn't this?

This is not a generic AI demo, and it is not busywork.

### 2-minute theory

- The system request provides a framework. This will make the result predictable.
- Fewer but more precise instructions typically give better results.
- Quality comes from ensuring that role-based system instructions is verifiable and repeatable.

### Key concepts

- **Working framework:** a short guide that describes the three objectives, format and limits.
- **Quality gate:** quick check before accepting the output.

## Where

### It works fine here

- handling several types of tasks in a uniform tone in situations where a quick and clear decision is required
- when a role-based system instruction gives a clear next step to several people

### Do not use this when

- if you cannot cover the risk of inconsistent response quality with human control

### Connection points

- daily handovers between office manager and team assistant
- short statuses and decision preparation notes
- repetitive tasks, where a consistent output counts for 4 types of tasks

## When

### Use it when

- handling several types of tasks, you meet tasks in a uniform tone, and time is short
- based on role-based system instructions, several actors must act in the same way

### Frequency

At least once per workday, and every time for recurring tasks.

### Late warning signs

- role-based system instructions need to be rewritten every time
- you often get back that the content does not handle inconsistent response quality problem

## Why it matters

### Practical benefits

- a role-based system instruction is created faster, with fewer rounds
- quality will be more measurable with a consistent output indicator for 4 task types
- rework is reduced, because you filter out inconsistent response quality errors early

### What happens if you skip it

- the number of misunderstandings increases when handling several types of tasks in a uniform tone
- it is more difficult to maintain a consistent output target for 4 task types

### Realistic expectation
- It will improve: the practical application of assistant system requests and the quality of the transferable output.
- Not guaranteed: infallible response without human control in every run.

## How

### Step by step

1. Name the work situation and the expected result exactly. 2. Write down the mandatory limits (time frame, scope, style, prohibitions). 3. Request the first AI version in a specific format. 4. Run a 3-point quality gate, paying particular attention to **inconsistent response quality** risk. 5. Fix the output, then save it as a usable template.

### Do / Don't

**Do**
- Enter a short, specific task description so that the role-based system command format is clear from the start.
- Before finalizing, check whether consistent output measurement points have been met for 4 task types.


**Don't**
- Don't leave the topic of inconsistent response quality in the background, even if the response seems good at first.
- Do not ask for a general text if a specific decision requires the handling of several task types in a uniform tone for a task.

### Common errors and their correction

- **Error:** too general a request for the handling of several types of tasks in a uniform tone. **Correction:** name the person concerned, the deadline and the decision target.
- **Error:** there is no connection between output and consistent output measurement for 4 task types. **Fix:** record 1 binary pass/fail result at the end of each run.

### It's ready if

- [ ] You have completed and saved **role-based system instruction** output.
- [ ] You checked the output according to at least 3 quality criteria.
- [ ] You have documented the **consistent output for 4 task types** indicator.

## Guided exercise

### Inputs

- a short description of a real task about the following situation: **Handling several types of tasks in a uniform tone**
- a short own sample that shows what you consider a good assistant system request output
- checklist to exclude the risk of inconsistent response quality

### Steps

1. Write in 4 lines the situation, the goal and how you will measure consistent output results for 4 task types.
2. Create a sample request to generate a role-based system statement, then request two different versions.
3. Compare the versions in the table and mark which one can go on for real use.

### Expected output format

| Element | What do you type | Expected result |
| --- | --- | --- |
| Situation | Handling several task types in a uniform tone | Everyone involved understands from one reading |
| Request frame | Target + context + format + constraint | Minimum of 2 usable versions |
| Quality gate | Accuracy + Risk + Next Step | Clear repair decision |
| Final output | role-based system command | Saved, reusable template |

> **Common mistake:** Many people only look at the risk of inconsistent answer quality at the end, even though it should already be indicated in the task description.
