# Lesson 11: Creating your own request directory

**In one sentence:** A good request is not a one-time idea, but a reusable tool.
**Time:** 20-30 minutes
**Tangible output:** tagged request library v1

## Learning goal

Weekly customer communication tasks. In this situation, you are a sales support specialist and you have to make a quick but reliable decision.
By the end of today's practical task, you will have a labeled request library v1 in your hand, which you can use again tomorrow.
Key insight: the request library starts working when you make the output measurable at startup. At the end of the lesson, you will be able to: **build quickly retrievable requests for recurring tasks**

### Success criteria (observable)
- [ ] The specific daily output is ready: **tagged request library v1**.
- [ ] At least one measurable indicator is attached to the output: **10 tagged requests saved and documented**.
- [ ] There is a separate control step for managing the main risk: **always start from zero**.

### Output you will create today
- **Name:** tagged request library v1
- **Format:** editable board and short decision note
- **Where saved:** `ai-rutin/nap-11-sajat-keres-konyvtar-letrehozasa.md`

## Who

**Primary Role:** Sales Support Specialist
**Secondary Role:** Sales Representative

## What

### What is this?

This lesson shows in practice how to apply the **request library** method in a real work situation.
The goal is not a theory, but an output that you can really use in your next working day.

### What isn't this?

This is not a generic AI demo, and it is not busywork.

### 2-minute theory

- The library increases speed and quality at the same time if you maintain it.
- Quality requires two things: a good task description and consistent control.
- Quality comes from ensuring that tagged request library v1 is verifiable and repeatable.

### Key concepts

- **Working framework:** a short guide that describes the three objectives, format and limits.
- **Quality gate:** quick check before accepting the output.

## Where

### It works fine here

- weekly recurring customer communication tasks in situations where a quick and clear decision is required
- when tagged request library v1 gives clear next step to multiple people

### Do not use this when

- if you cannot always cover the risk of starting from scratch with human control

### Connection points

- daily handovers between sales support specialist and sales representative
- short statuses and decision preparation notes
- repetitive tasks, where 10 tagged requests saved and documented count

## When

### Use it when

- you encounter weekly customer communication tasks and you have little time
- based on tagged request library v1, several actors must act in the same way

### Frequency

At least once per workday, and every time for recurring tasks.

### Late warning signs

- tagged request library v1 needs to be rewritten every time
- you often get back that the content does not always handle the problem of starting from scratch

## Why it matters

### Practical benefits

- tagged request library v1 is created faster, with fewer rounds
- quality will be more measurable 10 tagged requests saved and documented with indicators
- the rework is reduced, because you always filter out the error of starting from scratch early

### What happens if you skip it

- the number of misunderstandings increases in the case of weekly customer communication tasks
- harder to keep 10 tagged requests saved and documented goal

### Realistic expectation
- It will improve: the practical application of the request library and the quality of the transferable output.
- Not guaranteed: infallible response without human control in every run.

## How

### Step by step

1. Name the work situation and the expected result exactly. 2. Write down the mandatory limits (time frame, scope, style, prohibitions). 3. Request the first AI version in a specific format. 4. Run a 3-point quality gate, paying particular attention to **always starting from scratch** risk. 5. Fix the output, then save it as a usable template.

### Do / Don't

**Do**
- Enter a short, specific task description so that the v1 format of the tagged request library is clear from the start.
- Before finalizing, check whether 10 tagged requests have been saved and documented as measuring points.


**Don't**
- Don't always leave the topic of starting from scratch in the background, even if the answer seems good at first.
- Don't ask for a general text if you need material for a specific decision for weekly recurring customer communication tasks.

### Common errors and their correction

- **Error:** an overly general request is initiated for weekly recurring customer communication tasks. **Correction:** name the person concerned, the deadline and the decision target.
- **Error:** no connection between output and 10 tagged requests saved and documented measurement. **Fix:** record 1 binary pass/fail result at the end of each run.

## Guided exercise

### Inputs

### Steps
