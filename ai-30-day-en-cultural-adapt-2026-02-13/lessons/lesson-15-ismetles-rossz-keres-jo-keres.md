# Lesson 15: Repetition: bad request → good request **In a sentence:** Even a bad request can be good if you use conscious iteration.
**Time:** 20-30 minutes
**Tangible output:** 2-round request rewrite frame

## Learning goal

Creating a usable draft from a rough request. In this situation, you are responsible for the content and you have to make a quick but reliable decision.
By the end of today's practical task, you will have a 2-round request rewriting framework, which you can use again tomorrow.
Key insight: request rewriting starts working when you make the output measurable at the start. At the end of the lesson you will be able to: **fix a weak request step by step**

- [ ] At least one measurable indicator is attached to the output: **measurable quality improvement after 2 iterations**.
- [ ] There is a separate control step to manage the main risk: **repair that stops after the first round**.

### Output you will create today
- **Name:** 2-round request rewrite frame
- **Format:** editable board and short decision note

## What

### What is this?

This lesson shows in practice how to apply the **request rewrite** method in a real work situation.
The goal is not a theory, but an output that you can really use in your next working day.

### What isn't this?

This is not a generic AI demo, and it is not busywork.

### 2-minute theory

- The goal of iteration is not more text, but better decision quality.
- Quality requires two things: a good task description and consistent control.
- Quality comes from ensuring that the 2-round request rewriting frame is verifiable and repeatable.

### Key concepts

- **Working framework:** a short guide that describes the three objectives, format and limits.
- **Quality gate:** quick check before accepting the output.

## Where

### It works fine here

- it can be used to create a draft from a rough request in situations where a quick and clear decision is needed
- when a 2-round request rewrite frame gives a clear next step to several people

### Do not use this when

- if you cannot cover the risk of repairs stopping after the first round with human inspection

### Connection points

- daily handovers between content manager and editor
- short statuses and decision preparation notes
- repetitive tasks, where measurable quality improvement after 2 iterations counts

## When

### Use it when

- you encounter the task of creating a usable draft from a rough request, and you have little time
- Based on a 2-round request rewrite frame, several actors must act in the same way

### Frequency

At least once per workday, and every time for recurring tasks.

### Late warning signs

- 2-round request rewrite frame needs to be rewritten every time
- you often get back that the content does not handle the repair problem that stops after the first round

## Why it matters

### Practical benefits

- a 2-round request rewrite frame is completed faster, with fewer rounds
- quality will be more measurable with a quality improvement indicator that can be measured after 2 iterations
- the rework is reduced, because you filter out early a repair error that stops after the first round

### What happens if you skip it

- the number of misunderstandings increases in the situation of creating a usable draft from a rough request
- it is more difficult to maintain a measurable quality improvement goal after 2 iterations

### Realistic expectations

- It will improve: the practical application of request rewriting and the quality of the deliverable output.
- Not guaranteed: infallible response without human control in every run.

## How

### Step by step

1. Name the work situation and the expected result exactly. 2. Write down the mandatory limits (time frame, scope, style, prohibitions). 3. Request the first AI version in a specific format. 4. Run a 3-point quality gate, paying special attention to the risk of **stopping repair after the first round**. 5. Fix the output, then save it as a usable template.

### Do / Don't

**Do**
- Enter a short, specific task description so that the format of the 2-round request rewrite frame is clear from the beginning.
- Before finalizing, check whether the quality improvement measurement point that can be measured after 2 iterations has been met.


**Don't**
- Don't leave the topic of correction after the first round in the background, even if the answer seems good at first.
- Don't ask for a general text if you need a specific decision for the task of creating a usable draft from a rough request.

### Common errors and their correction

- **Error:** too general a request starts from a rough request for a usable draft situation. **Correction:** name the person concerned, the deadline and the decision target.
- **Error:** there is no connection between the output and the quality improvement measurement that can be measured after 2 iterations. **Fix:** record 1 binary pass/fail result at the end of each run.
