# Lesson 10: Replay and Request Debugging Day **In a sentence:** A bad request can be fixed if you know what you're looking at.
**Time:** 20-30 minutes
**Tangible output:** request error catalog with correction

## Learning goal

Recurring poor AI output in a team. In this situation, you are a product manager and you have to make a quick but reliable decision.
At the end of today's practical task, you will have a request with a bug catalog correction, which you can use again tomorrow.
Key insight: request debugging starts working when you make the output measurable at startup. At the end of the lesson, you will be able to: **recognize and correct typical request errors**

- [ ] The specific daily output is ready: **request error catalog with correction**.
- [ ] At least one measurable indicator is attached to the output: **3 corrected versions of 3 incorrect requests**.
- [ ] There is a separate control step to deal with the main risk: **the same error is repeated**.

- **Name:** request error catalog with correction
- **Format:** editable board and short decision note

## Who

**Primary Role:** Product Manager
**Secondary role:** project coordinator
**Involved decision maker:** Product Director

## What

### What is this?

This lesson shows in practice how to apply the **request debugging** method in a real work situation.
The goal is not a theory, but an output that you can really use in your next working day.

### What isn't this?

This is not a generic AI demo, and it is not busywork.

### 2-minute theory

- We correct the error not based on feeling, but on the basis of symptoms: goal, context, form, limitation.
- Fewer but more precise instructions typically give better results.
- Quality comes from ensuring that the request can be checked and repeated by correcting the catalog of errors.

### Key concepts

- **Working framework:** a short guide that describes the three objectives, format and limits.
- **Quality gate:** quick check before accepting the output.

## Where

### It works fine here

- recurrently poor ai output in team-type situations where a quick and clean decision is needed
- when a request error catalog is corrected, it gives a clear next step to several people

### Do not use this when

- if the same error is repeated, you cannot cover the risk with human control

### Connection points

- daily handovers between product manager and project coordinator
- short statuses and decision preparation notes
- repetitive tasks, where 3 incorrect requests count as 3 corrected versions

## When

### Use it when

- recurring poor AI output you encounter a task in a team and time is short
- based on request error catalog with correction, several actors must act in the same way

### Frequency

At least once per workday, and every time for recurring tasks.

### Late warning signs

- request error catalog needs to be rewritten every time with corrections
- you often get back that the content doesn't handle the same error repeating problem

## Why it matters

### Practical benefits

- the request is completed faster with error catalog correction, with fewer rounds
- the quality will be more measurable with an indicator of 3 corrected versions out of 3 incorrect requests
- the rework is reduced, because you filter out the same error repeatedly

### What happens if you skip it

- the number of misunderstandings increases, recurring poor AI output in a team situation
- it is more difficult to keep 3 corrected version targets out of 3 incorrect requests

### Realistic expectations

- It will improve: the practical application of request debugging and the quality of the transferable output.
- Not guaranteed: infallible response without human control in every run.

## How

### Step by step

1. Name the work situation and the expected result exactly. 2. Write down the mandatory limits (time frame, scope, style, prohibitions). 3. Request the first AI version in a specific format. 4. Run a 3-point quality gate, paying special attention to the risk of **the same mistake being repeated**. 5. Fix the output, then save it as a usable template.

### Do / Don't

**Do**
- Enter a short, specific task description so that the format of the error catalog with corrections of the request is clear from the start.
- Before finalizing, check whether 3 corrected version measurement points out of 3 incorrect requests have been fulfilled.


**Don't**
- Don't leave the topic of repeating the same error in the background, even if the answer seems good at first.
- Don't ask for a general text if you need material for a specific decision, recurrently weak AI output for a task in a team.

### Common errors and their correction

- **Error:** overly general request repeatedly starts with weak ai output for a situation in a team. **Correction:** name the person concerned, the deadline and the decision target.
- **Error:** there is no connection between the output and 3 corrected version measurements out of 3 faulty requests. **Fix:** record 1 binary pass/fail result at the end of each run.

- [ ] Completed and saved **request error catalog with correction** output.
- [ ] You checked the output according to at least 3 quality criteria.
- [ ] You have documented the indicator **3 corrected versions out of 3 incorrect requests**.

## Guided exercise

### Inputs

- a brief description of a real task for the following situation: **Recurring poor AI output in a team**
- a short own sample that shows what you consider a good request debugging output
- checklist to exclude the risk that the same error is repeated

### Steps

1. Write in 4 lines the situation, the goal and how you will measure the results of 3 corrected versions from 3 incorrect requests.
2. Create a sample request to generate a request error catalog with corrections, then request two different versions.
3. Compare the versions in the table and mark which one can go on for real use.

### Expected output format

| Element | What do you type | Expected result |
| --- | --- | --- |
| Situation | Recurring poor AI output in a team | Everyone involved understands from one reading |
| Request frame | Target + context + format + constraint | Minimum of 2 usable versions |
| Quality gate | Accuracy + Risk + Next Step | Clear repair decision |
| Final output | request error catalog with correction | Saved, reusable template |

> **Common mistake:** many people only look at the risk of repeating the same error at the end, even though it should already be indicated in the task description.
